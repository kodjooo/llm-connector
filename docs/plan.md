# План реализации LLM gateway для site classification

## Общие принципы

- Запуск, разработка, тестирование и проверка сервиса выполняются только через Docker Desktop.
- Локальное Python-окружение вне контейнеров не используется.
- Все изменения должны сопровождаться актуализацией `docs/architecture.md`, `.env.example`, `.env`, тестов в `tests` и при необходимости `README.md`.
- После завершения каждого этапа в этом документе рядом с этапом нужно дописывать `выполнено`, не удаляя сам этап.

## Этап 1. Подготовка каркаса сервиса

- Статус: выполнено
- Цель: превратить текущий шаблон проекта в каркас HTTP-сервиса для gateway.
- Задачи:
- Выбрать и зафиксировать FastAPI как основной фреймворк сервиса.
- Пересобрать структуру приложения под веб-сервис: точки входа API, конфигурация, схемы, сервисный слой, клиенты, middleware.
- Подключить базовые зависимости в `requirements.txt`.
- Подготовить `docker-compose.yml` как основной способ запуска проекта через Docker Desktop.
- Обновить `Dockerfile` под запуск ASGI-приложения.
- Подготовить `.env.example` и `.env` с обязательными переменными:
- `OPENAI_API_KEY`
- `GATEWAY_AUTH_TOKEN`
- `DEFAULT_OPENAI_MODEL`
- `LOG_LEVEL`
- при необходимости дополнительные переменные для timeout, retry, payload limit, rate limit, Sentry.
- В `.env` добавить комментарии, какой доступ нужен и где брать значения.
- Результат этапа:
- проект поднимается через Docker Desktop как HTTP-сервис;
- есть базовый контейнер приложения;
- есть единая конфигурация окружения.

## Этап 2. Базовые endpoint и контракт API

- Статус: выполнено
- Цель: реализовать обязательные endpoint и строгую валидацию входного и выходного контракта.
- Задачи:
- Реализовать `GET /health` с ответом `{"status":"ok"}`.
- Реализовать `POST /v1/site-classification`.
- Описать Pydantic-схемы для входного payload:
- проверка `schema`;
- проверка вложенных структур `serp`, `serp_screening`, `homepage_screening`;
- поддержка nullable-полей;
- ограничение допустимых типов и длины строк.
- Описать нормализованную выходную схему ответа.
- Зафиксировать допустимые значения `site_verdict`.
- Добавить единый формат ошибок:
- `400` для невалидного запроса и неизвестной схемы;
- `401` для невалидной авторизации;
- `429` для локального rate limit;
- `502` для невалидного/необработанного ответа OpenAI;
- `504` для таймаута upstream.
- Результат этапа:
- сервис принимает запросы по заданному контракту;
- некорректные запросы предсказуемо отклоняются.

## Этап 3. Конфигурация, безопасность и middleware

- Статус: выполнено
- Цель: закрыть базовые требования по защите и управлению запросами.
- Задачи:
- Реализовать проверку `Authorization: Bearer <token>` через `GATEWAY_AUTH_TOKEN`.
- Добавить middleware/request hooks для:
- генерации или проброса `request_id`;
- измерения длительности обработки;
- безопасного логирования запроса без чувствительных данных.
- Ограничить максимальный размер входного payload до 32 KB.
- Настроить маскирование чувствительных данных в логах:
- не писать `Authorization`;
- не писать полный `homepage_excerpt`;
- в debug-режиме логировать не более 500 символов excerpt.
- Подготовить конфигурацию для ограничения доступа по сети:
- документировать работу за reverse proxy/private network/VPN;
- при возможности предусмотреть доверенный список IP через конфигурацию.
- Результат этапа:
- сервис защищен bearer-токеном;
- соблюдены ключевые требования к логированию и ограничению входных данных.

## Этап 4. Интеграция с OpenAI и нормализация ответа

- Статус: выполнено
- Цель: реализовать основной бизнес-процесс обращения к OpenAI и возврата стандартизированного результата.
- Задачи:
- Подключить клиент OpenAI через официальный SDK.
- Реализовать выбор модели:
- из поля `model` запроса, если оно передано;
- из `DEFAULT_OPENAI_MODEL`, если поле отсутствует.
- Сформировать системный и пользовательский prompt для site classification на основе входного payload.
- Настроить timeout одного запроса в 45 секунд.
- Реализовать retry на временных ошибках OpenAI: до 3 попыток с backoff 1s, 2s, 3s.
- Реализовать парсинг ответа модели без возврата сырого OpenAI response клиенту.
- Провалидировать результат:
- `site_verdict` только из разрешенного списка;
- `confidence` приводится к `float`;
- `reason`, `detected_city`, `provider`, `model` нормализуются.
- Возвращать `502`, если ответ OpenAI не удается распарсить или он не соответствует контракту.
- Возвращать `504` при превышении timeout upstream.
- Результат этапа:
- gateway выполняет классификацию через OpenAI и возвращает только нормализованный JSON.
